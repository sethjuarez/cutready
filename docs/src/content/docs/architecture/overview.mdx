## Frontend

The frontend is a **React 19** single-page application bundled with **Vite**:

| Technology | Purpose |
|-----------|---------|
| **React 19** | UI component framework |
| **TypeScript** | Type-safe frontend code |
| **Zustand** | Centralized state management |
| **Tailwind CSS** | Utility-first styling |
| **CodeMirror** | Markdown editing (notes) |
| **@dnd-kit** | Drag-and-drop (storyboard list, table rows) |
| **react-markdown** | Markdown preview rendering |

### State Management

All application state flows through a single **Zustand store** (`appStore.ts`):
- Project state (current project, recent list)
- Document state (sketches, storyboards, notes)
- UI state (sidebar, tabs, panels, theme)
- Recording state (profiles, active session)
- Version history (snapshots, timeline, dirty flags)

The store dispatches **Tauri commands** to the Rust backend for all
filesystem and system operations.

## Backend (Rust)

The Rust backend handles:

### Commands Layer (`src/commands/`)

| Module | Responsibility |
|--------|---------------|
| `project.rs` | Project CRUD, recent projects, sidebar persistence |
| `sketch.rs` | Sketch document CRUD, list summaries |
| `storyboard.rs` | Storyboard CRUD, section management |
| `note.rs` | Markdown note CRUD |
| `interaction.rs` | Browser profile detection, recording sessions |
| `screenshot.rs` | Monitor detection, screen/region capture |
| `versioning.rs` | Snapshot save/restore, timeline, stash |

### Engine Layer (`src/engine/`)

Lower-level implementations:
- **interaction.rs** â€” CDP browser communication, Playwright sidecar management
- **screenshot.rs** â€” Native screen capture via `xcap` crate
- **versioning.rs** â€” Git operations via `gix` (gitoxide)

### Models Layer (`src/models/`)

Shared data types:
- **action.rs** â€” Tagged union of browser/native actions
- **script.rs** â€” Script rows with metadata
- **session.rs** â€” Recording session with captured actions
- **sketch.rs** â€” Sketch, Storyboard, VersionEntry, GraphNode

## Playwright Sidecar

Browser automation runs in a **Node.js sidecar process** managed by Tauri:

1. The Rust backend spawns a Playwright Node.js process
2. Communication happens over stdio (JSON messages)
3. The sidecar connects to Chrome/Edge via CDP (Chrome DevTools Protocol)
4. Actions are streamed back to the Rust backend in real-time

This architecture isolates browser automation from the main process,
preventing UI freezes during recording.

## Build & Release

CutReady uses **GitHub Actions** for automated cross-platform builds:

- **macOS** â€” ARM (Apple Silicon) + Intel (x86_64) builds
- **Windows** â€” MSI + NSIS installers
- **Linux** â€” DEB + AppImage packages (Ubuntu 24.04)

Releases are triggered by pushing a `v*` git tag and create draft GitHub
releases with all platform artifacts.
